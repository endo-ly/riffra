# Riffra テスト戦略

## 1. 品質の考え方

テストは、Play、Design、Arrangeの制作循環を、必要以上の実行時間と手作業を使わずに守るために行う。

優先して検出する問題は次のとおり。

1. 危険な音声出力と緊急ミュートの不成立
2. 録音、素材、セッションの消失または破損
3. 失敗を成功として表示する状態
4. VST3、音声機器、MIDI、子プロセスの不整合
5. 保存後または再起動後に制作内容が変わる状態
6. 主要操作へ到達できない画面配置
7. 生成式が異常な値や予測できない負荷を音声処理へ持ち込む状態

テスト件数や網羅率そのものを目標にしない。利用者への影響が大きく、再発しやすい問題を、原因に近い層で検出する。

## 2. 検証の構成

下層を厚く、上層ほど代表的な制作フローへ絞る。

```text
              Windows実機での受入
        実音・実機器・実VST3・画面配置

             全体を通す簡易確認
        Play / Design / Arrange / 保存復元

                  結合テスト
      Tauri命令・子プロセス通信・ファイル保存

                 部品テスト
           React操作・表示・状態反映

                 単体テスト
       制作規則・変換・安全条件・純粋計算
```

| 層           | 主な境界                     | 確認する内容                           | 実行頻度           |
| ------------ | ---------------------------- | -------------------------------------- | ------------------ |
| 単体         | 関数、状態遷移、音声処理部品 | 計算、変換、不変条件、異常値、安全条件 | 常時               |
| 部品         | ReactとNative API契約        | 操作、命令発行、表示、失敗、到達性     | 常時               |
| 結合         | Rust、C++、Tauri、ファイル   | 通信、保存、録音情報、索引、子プロセス | 変更領域ごと       |
| 全体簡易確認 | アプリ全体                   | 三領域の代表フローと保存復元           | 作業単位の終端     |
| Windows実機  | OS、機器、VST3、実音         | 自動化で代替できない挙動               | マイルストーン終端 |

## 3. 基本原則

### 3.1 原因に近い層で確認する

同じ問題を複数の層で検出できる場合は、速く、原因を特定しやすい層を選ぶ。画面を通さなくても確認できる計算や状態遷移を、実機操作で繰り返さない。

ただし、データ消失、危険な音声、破壊的変更には、単体と結合など異なる境界で防御を重ねる。

### 3.2 利用者の挙動を検証する

ファイル名や内部関数の呼出し順ではなく、操作と結果を確認する。内部実装を置き換えても、期待結果が同じならテストを維持できる形を優先する。

### 3.3 不具合修正には再現テストを添える

自動化できる不具合は、修正前に失敗し、修正後に成功するテストを追加する。同じ種類の不具合を再び画面操作で探さない。

自動化できない場合は、条件、操作、期待結果、確認した証拠を挙動確認・課題管理表へ記録する。

### 3.4 実機確認をまとめる

関連する修正の自動テストが通るまでは実機確認へ進まない。同じ機器、プラグイン、録音素材、起動状態を使える項目をまとめ、マイルストーン終端で一度確認する。

### 3.5 維持費に見合うテストを残す

軽微な内部変更で頻繁に壊れ、重要な不具合を検出しないテストは減らす。色、余白、装飾だけの違いは原則として自動化しない。主要操作が隠れる、重なる、到達できない問題は検出対象とする。

## 4. 単体テスト

外部プロセス、実機器、利用者の制作ファイルへ依存しない処理を確認する。

### 4.1 TypeScript

- Session、Rack、Recording、Clip、Assetの状態遷移
- 入力値の正規化と表示用変換
- 音声状態に応じたミュートと操作可否
- 取消し、やり直し、音色候補の切替
- 生成定義の編集、選択、保存状態
- Native応答を画面状態へ反映する規則

### 4.2 Rust

- 保存形式の読込み、検証、更新
- 録音情報と素材索引の整合性
- Tauri命令へ渡す値の検証
- 音声子プロセス応答の解析と分類
- 分析、分離、書出しの純粋な計算
- 生成式の解析、値の範囲、接続の検証
- 乱数条件を固定した生成結果の再現性
- 生成定義と音声素材の由来
- 後処理の進捗、取消し、結果採用
- パス、容量、範囲、音量などの安全条件

### 4.3 C++

- 直流成分の除去
- フィードバックの疑いの検出
- 非数、無限値、過大音量の防止
- ラックの順序、迂回、状態変更
- 検証済み生成定義の実行とパラメーター変更
- ノート、強さ、離鍵、同時発音、発音停止の処理
- 生成処理の計算量、反復回数、内部状態の上限
- 録音の開始、途中確定、正常終了、異常終了
- 通信命令の解析

内部関数を直接呼ぶためだけに公開範囲を広げない。入力、結果、不変条件を公開された責務の単位で確認する。

## 5. 部品テスト

Reactを実機の音声処理から切り離し、利用者の操作と表示結果を確認する。

Native APIは本番と同じ契約を実装する差替えへ置き換え、次の状態を決定的に返せるようにする。

- 成功
- 失敗
- 処理中
- 取消し
- 部分成功
- 実効値が要求値と異なる状態
- 音声機器またはプラグインが利用できない状態

確認する内容:

- Play、Design、Arrangeで共有状態が一致する
- 操作に対応する命令だけが発行される
- 命令結果が画面とSessionへ反映される
- 失敗を成功表示へ変換しない
- 空、処理中、失敗、復旧可能を区別する
- キーボード操作と主要な読み上げ名がある
- スクロール領域内に主要操作が含まれる
- 無効な生成式の箇所と修正方法が分かる
- 編集中に失敗しても直前の有効な音と定義を失わない

差替え実装は、本番に存在しない成功経路を作らない。存在しない識別子、無効なパス、失敗した保存を成功として返さない。

## 6. 結合テスト

### 6.1 Rustと音声子プロセス

実際の`riffra-audio.exe`を起動し、標準入出力のJSON Linesで通信する。

確認する内容:

- 起動、状態取得、終了
- 要求識別子と応答の対応
- 再生、停止、録音、機器復旧の状態遷移
- プラグイン、音声機器、録音のエラー分類
- 不正な命令、存在しないファイル、日本語を含むパス
- 子プロセス異常終了の検出と再起動
- 破棄後にコールバックと子プロセスが残らないこと

通信境界を確認するため、ここでは音声子プロセスを偽物へ置き換えない。

### 6.2 Tauri命令

画面から送るJSONとRustの型、管理状態、戻り値、エラーの契約を確認する。

- 命令が登録されている
- 引数名と値を正しく読み取る
- 必要な管理状態を渡す
- 結果とエラーを画面が読める形で返す
- 制作規則を持つ処理へ正しく委譲する

Tauri命令は変換と委譲に限定し、主要な制作規則は単体テストで確認する。

### 6.3 ファイルと永続化

テスト専用の一時領域を使い、次を確認する。

- Sessionの自動保存、世代復旧、Project入出力
- 原音、加工音、MIDI、録音情報の整合性
- Inboxの名称変更、タグ、保管、昇格、削除
- ファイルとLibrary索引の一致
- 波形編集設定と派生素材の由来
- 生成式、接続、パラメーター、乱数条件、形式の版の保存と復元
- 生成定義から作った音声素材の由来
- 書出しの途中ファイルと完成ファイルの区別
- プラグイン走査結果と隔離情報

利用者のAppData、VST3フォルダー、制作ファイルへ書き込まない。

## 7. 制作フローの簡易確認

全体を通すテストは、分岐の網羅ではなく、層をまたぐ接続切れを検出する。

### 7.1 Play

1. Scratch Sessionを開く
2. テスト用の音声入出力を選ぶ
3. VST3またはテスト用処理をラックへ入れる
4. パラメーターを変更する
5. 迂回とミュートを切り替える
6. ラック状態を保存して読み直す

### 7.2 録音と素材

1. 原音と加工音の録音を開始・停止する
2. Inboxに録音が現れる
3. 試聴、名称変更、タグを行う
4. セッションを読み直して録音を参照できる

### 7.3 Design

素材から作る流れ:

1. 音声を選ぶ
2. 使用範囲とループを設定する
3. パッドまたは鍵盤へ割り当てる
4. 新しい素材として保存する
5. 元素材との関係を確認する

数式から作る流れ:

1. 固定した生成式、基本波形、乱数条件から音を作る
2. 変調、包絡、フィルターを接続する
3. MIDIノートと強さを変えて演奏する
4. 生成定義を保存して開き直す
5. 同じ入力で同じ音が再現されることを数値で確認する
6. 音声素材へ変換し、生成定義との関係を確認する

### 7.4 Arrange

1. 音声とMIDIを時間軸へ置く
2. 移動、分割、ループを行う
3. 再生する
4. WAVを書き出す
5. 元素材が変更されていないことを確認する

### 7.5 保存と再開

1. 三領域の状態を含むProjectを保存する
2. アプリを終了する
3. Projectを開き直す
4. 素材、ラック、時間軸、設定を復元する
5. 不足項目がある場合も残りを開く

## 8. Windows実機での受入

実機でしか判断できない内容に限定する。

### 8.1 Play

- WASAPI、ASIO、実際のオーディオインターフェース
- 楽器とマイクの入力
- 入出力チャンネル、遅延、音切れ、ノイズ
- 実VST3の走査、読込み、音声処理、編集画面、状態復元
- MIDI機器の接続、切断、発音停止
- 緊急ミュートと音量変化

### 8.2 DesignとArrange

- 実際に聞こえる切れ目、クリック、ループのつながり
- 生成音の発音、離鍵、音程変化、同時発音の聴感
- 生成定義の編集中に音切れや危険な音量が発生しないこと
- 試聴と再生の音量差
- 波形編集画面と時間軸の到達性
- 書き出した音声の実再生

### 8.3 Windowsと外部境界

- 表示倍率、複数画面、プラグイン画面の配置
- 子プロセス終了、孤立、ファイルロック
- 長いパス、日本語を含むパス、権限
- 音声機器の抜去と再接続
- 第三者VST3の異常時の隔離
- 長時間再生・録音と資源使用量

物理障害や第三者プラグインの異常は、通常の機能開発ごとに繰り返さず、リリース品質の確認へまとめる。

## 9. 障害と差替えの扱い

障害を自動化するのは、次のいずれかに当てはまる場合とする。

- データ消失または危険な音声につながる
- 失敗を成功として表示する可能性がある
- 保存と再起動で状態が変わる
- 過去に再発した
- 手作業では組合せを見落としやすい

すべての物理障害を開発中に再現する必要はない。まず、音声を止める、取得済みデータを残す、失敗を表示する、再試行できる、という共通の安全条件を自動テストで守る。

大規模な共通障害基盤を先に作らず、必要な制作フローで実際に使う失敗から小さく差し替える。

## 10. テストデータ

- 短いPCM WAV、MIDI、録音情報、Sessionを使う
- 時刻、乱数、識別子は固定または差替え可能にする
- 生成式は正常、構文不正、非有限値、計算上限超過を最小例で用意する
- 第三者VST3本体と利用者の制作物を含めない
- 大容量音声と長時間試験を通常テストから分離する
- 生成物はテスト専用領域へ置き、終了時に片付ける
- 音声比較はサンプル数、ピーク、実効値、ハッシュ、許容誤差を目的に応じて使う

## 11. 完了条件

不具合修正と機能実装は、次を満たした時点で完了する。

- 対象の利用者フローと期待結果が明確である
- 原因と影響範囲を説明できる
- 自動化可能な不具合に回帰テストがある
- 対象に近いテストと関連する既存テストが成功する
- 保存データと外部プロセスを壊していない
- 実機依存項目をWindows実機で確認している
- 挙動確認・課題管理表に判定と証拠がある

ビルド成功、型検査成功、画面要素の存在だけでは、利用者の挙動が成立したことを意味しない。
