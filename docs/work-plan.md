# Riffra 作業計画

## 1. 作業の基本方針

実装順は技術領域の網羅ではなく、利用者が完遂できる制作体験を基準に決める。

優先順位は次のとおり。

1. データ消失と危険な音声出力を防ぐ
2. 見えている不具合を直す
3. Playで実際に演奏できる状態を作る
4. 演奏を録音し、素材として再利用できる状態を作る
5. Designで素材または数式から音源を作れる状態を作る
6. Arrangeで音を組み合わせ、書き出せる状態を作る
7. 制作循環を保存・再開できる状態を作る
8. 補助機能とリリース品質を整える

画面、命令、保存処理の一部だけを完成させても、利用者の操作が最後までつながらなければ完了としない。

## 2. 作業単位

一つの作業単位は、同じ利用者フローと保存結果を共有する複数の要求で構成する。

| 種類             | 扱い                                             |
| ---------------- | ------------------------------------------------ |
| 既存の不具合     | 同じ作業単位の新規実装より先に直す               |
| 未実装           | 必要な下位機能と一緒に実装する                   |
| 検証しにくい構造 | 対象フローに必要な範囲だけ分離する               |
| 実機依存         | 自動検証を終えてから、関連項目をまとめて確認する |
| 外部条件待ち     | 管理表へ条件を明記し、別の作業へ進む             |

同じ型、通信方式、保存形式、音声コールバックを複数の作業単位から同時に変更しない。

## 3. 一つの作業単位の進め方

### 3.1 成立条件を定める

1. 対象の要求IDを選ぶ
2. 利用者の開始状態、操作、期待結果を一つの流れにする
3. 音、画面、保存データ、履歴、外部機器の観測点を決める
4. 正常系と、利用者が遭遇しやすい代表的な失敗系を決める

### 3.2 実効経路を確認する

1. Reactの操作からTauri命令までを追う
2. Rustから音声子プロセス、ファイル、索引までを追う
3. 画面だけ存在する箇所、途中まで接続された箇所、未実装を分ける
4. 保存と再起動を含めた断点を列挙する

### 3.3 修正・実装する

1. 再現できる不具合には回帰テストを置く
2. 同じ原因の不具合をまとめて直す
3. 利用者フローを成立させるために不足する機能を実装する
4. 保存形式と公開契約の変更を確認する
5. 不要になった迂回処理と重複処理を整理する

### 3.4 検証する

1. 対象に近い単体・部品テストを実行する
2. 通信、保存、ファイルの結合を確認する
3. 作業単位の終端で非GUI検証をまとめて実行する
4. 実機でしか分からない項目だけを一度確認する
5. 保存後と再起動後の状態を確認する
6. 管理表へ判定と証拠を反映する

## 4. リファクタリングの境界

リファクタリングは、制作フローの検証速度と不具合の局所化を改善する場合に行う。

先に分離する価値がある状態は次のとおり。

- 一つのReact部品が表示、状態遷移、Native呼出し、保存を同時に持つ
- 実機器や実プラグインがないと状態遷移を確認できない
- 成功、失敗、取消し、部分成功を差し替えられない
- 同じ検証や正規化が複数箇所にある
- 一つの変更が無関係な画面とテストを壊す

優先する分割単位は次のとおり。

- 制作規則を表す純粋な関数
- 機能単位の状態と操作
- Native APIの契約と実装
- 小さな表示部品
- ファイル、時刻、識別子、子プロセスの境界

全面的な書き直し、要求と無関係な名称変更、将来の可能性だけを理由にした抽象化は行わない。

## 5. マイルストーン

### M0. 検証基盤と構造整理

後続作業を速く、再現可能にするための最小基盤を整える。

成果:

- ReactとNative APIを差し替えて確認できる
- Rust、C++、TypeScriptの単体テストを実行できる
- 短いWAV、MIDI、録音情報、セッションを再利用できる
- 音声子プロセスとの通信を自動確認できる
- 失敗、取消し、部分成功を決定的に再現できる

検証基盤そのものを長期間の目的にせず、後続の制作フローで必要になった分だけ拡張する。

### M1. 安全性とデータ保全

制作機能を広げても、危険な音声出力と保存済みデータの消失を起こさない基盤を整える。

対象要求:

- `G-003`〜`G-006`
- `FLOW-007`
- `AUD-004`、`AUD-006`
- `REC-004`
- `PRJ-002`、`PRJ-004`
- `REL-001`〜`REL-004`
- `SEC-001`、`SEC-003`
- `DONE-002`

成果:

- 起動、機器変更、復旧直後は安全にミュートされる
- 異常な音声値と過大音量を出力しない
- 失敗を成功として表示しない
- 録音途中の原音と加工音を可能な範囲で残す
- 有効なセッション世代から復旧できる
- プラグインやファイルが不足しても残りの制作内容を開ける
- 子プロセスと後処理を終了・再実行できる

実機器の抜去、実記憶装置の故障、第三者プラグインのクラッシュを毎回の通過条件にはしない。通常経路の安全性を自動確認し、物理環境でしか判断できない最終確認はM7へまとめる。

### M2. Playの中核

楽器、声、外部音源、MIDIを入力し、VST3を通した音を低遅延で演奏できる状態を作る。

対象要求:

- `G-001`、`G-002`
- `FLOW-001`、`FLOW-002`
- `AUD-001`〜`AUD-003`、`AUD-005`
- `MIDI-001`〜`MIDI-003`
- `PLG-001`〜`PLG-005`
- `RACK-001`〜`RACK-006`
- `Q-002`

最初の完成フロー:

1. オーディオインターフェースの入力と出力を選ぶ
2. 入力メーターで楽器または声を確認する
3. VST3を検索してラックへ読み込む
4. 実際のプラグイン画面を開く
5. 演奏しながら音を変更する
6. 迂回、ミュート、音量変更を実音へ反映する
7. 音色を保存して比較する
8. アプリを開き直して同じ音へ戻る

ギターでは、AmpliTubeなどのアンプシミュレーターを開き、演奏可能な遅延で加工音を聞けることを代表的な完成像とする。

### M3. 録音と素材化

Playで作った音を録音し、失わず、聞き直し、次の制作へ渡せる状態を作る。

対象要求:

- `FLOW-003`
- `REC-001`〜`REC-003`
- `LIB-003`

完成フロー:

1. 原音と加工音を同時に録音する
2. 録音条件とMIDIを保存する
3. Inboxで録音を試聴する
4. 名称、タグ、保管先を変更する
5. 複数の録音候補を比較して採用する
6. セッションを開き直して録音と関連情報を確認する

### M4. Design

既存の素材を加工する方法と、数式から信号を生成する方法の両方で、新しい素材と演奏可能な音源を作れる状態にする。

対象要求:

- `FLOW-005`、`FLOW-008`
- `SYN-001`〜`SYN-005`
- `SMP-001`〜`SMP-006`
- `ANL-001`〜`ANL-003`
- `SEP-001`、`SEP-002`
- `LIB-001`、`LIB-002`、`LIB-004`

素材から作る流れ:

1. InboxまたはLibraryから素材を選ぶ
2. 波形を確認し、使用範囲を決める
3. 切出し、ループ、包絡、ピッチなどを非破壊で設定する
4. 必要に応じて分析、参照比較、音源分離を行う
5. パッドまたは鍵盤へ割り当てる
6. Playで演奏して反応を確かめる
7. 新しい素材として保存し、元素材との関係を辿る

数式から作る流れ:

1. 数式、基本波形、ノイズから音の出発点を作る
2. 生成器、変調、包絡、フィルター、音声処理を接続する
3. 無効な式、異常値、過大音量を安全に止める
4. PlayでMIDIから演奏し、音程、強さ、発音停止への反応を調整する
5. 生成式、接続、パラメーター、乱数条件を音色として保存する
6. 必要に応じて音声素材へ変換し、生成定義との関係を辿る

### M5. Arrangeと書出し

PlayとDesignで作った音を組み合わせ、音楽として確認し、外部へ渡せる状態を作る。

対象要求:

- `FLOW-004`
- `ARR-001`〜`ARR-005`
- `EXP-001`、`EXP-002`

完成フロー:

1. 音声とMIDIを時間軸へ置く
2. 元素材を変えずに移動、分割、範囲変更、ループを行う
3. トラックとミキサーの状態を一致させる
4. テンポ、拍子、マーカーを設定する
5. マスター、トラック、ステムを書き出す
6. WAVとMIDIを外部DAWへ同期位置を保って渡す

### M6. 制作循環の保存と支援

Play、Design、Arrangeを往復した制作内容を、検索、履歴、持出し、支援機能まで含めて継続できる状態を作る。

対象要求:

- `FLOW-006`
- `AI-001`〜`AI-005`
- `PRJ-001`、`PRJ-003`、`PRJ-005`、`PRJ-006`
- `WIN-002`

完成フロー:

1. 三領域を使った制作内容を一つのProjectへ保存する
2. 変更を取り消し、やり直す
3. 素材と制作内容を持ち運べる形へ収集する
4. 保存形式が異なる場合も元データを守る
5. AIの説明と提案を確認し、選択した変更だけを適用する
6. 外部送信の対象、目的、送信先を確認する

### M7. 統合品質と完成判定

Windows実機で制作循環を通し、長時間利用、表示、物理環境、外部機能を含む製品品質を確認する。

対象要求:

- `G-007`
- `WIN-001`
- `Q-001`、`Q-003`
- `DONE-001`

確認する制作循環:

1. Playで実際の機器とVST3を使って音を作る
2. 原音と加工音を録音する
3. Designで素材から演奏可能な音源を作る
4. Designで数式から別の音源を作り、Playで演奏する
5. Arrangeで二つの音源を組み合わせ、書き出す
6. 保存、終了、再起動後に同じ制作内容と音へ戻る

横断確認:

- 実際の音声機器とMIDI機器
- 第三者VST3の画面、状態復元、異常時の隔離
- 機器切断、記憶装置障害、セーフモード
- 表示倍率、複数画面、最小ウィンドウ
- 起動時間、長時間再生・録音、CPU、メモリー、ハンドル
- 不足ファイル、持出し、外部DAWへの受渡し

## 6. 実機確認のまとめ方

実機確認は小さな修正ごとに行わず、同じ機器、素材、プラグイン、起動状態を共有できる項目をまとめる。

一回の確認では次の順に観測する。

1. 起動と復旧状態
2. 音声機器、MIDI、ミュート、メーター
3. 対象フローの正常な操作
4. 代表的な失敗と利用者の復旧操作
5. 音声ファイル、保存情報、セッション、索引
6. 再起動後の復元

実機へ残すのは、実音、遅延、機器、VST3画面、OSのウィンドウ、子プロセス、ファイルロックなど、自動テストで代替できない事項だけとする。

## 7. 複数担当での分担

### 7.1 分担できる単位

- 制作規則と単体テスト
- Reactの表示部品と操作
- Rustの保存・索引処理
- C++の音声処理と自己検査
- テスト用素材と差替え実装
- 異なる制作領域の独立した機能

### 7.2 同時に変更しない単位

- 同じ通信形式の送信側と受信側
- Session型と保存形式の更新
- 同じ音声コールバック
- 同じ録音の開始・終了処理
- App全体の状態管理
- 同じ管理表行

### 7.3 引き渡す情報

- 対象要求IDと利用者フロー
- 確認した不具合と未実装
- 原因と影響範囲
- 変更した契約、保存形式、主要ファイル
- 追加したテストと結果
- 実行していない高価な検証
- 実機で行う操作と期待結果
- 残る依存関係

ビルド成功やテスト件数だけでなく、利用者がどこまで操作を完遂できるかを伝える。

## 8. マイルストーンの完了条件

マイルストーンは次をすべて満たしたときに完了する。

- 定義した完成フローを開始から終了まで実行できる
- 対象要求の既知の不具合を残していない
- 未実装として残す項目と理由が明確である
- 修正した不具合に回帰テストがある、または自動化しない理由がある
- 単体、部品、結合、必要なNative自己検査が成功する
- 保存、再起動、音声安全、子プロセス終了を確認している
- 実機依存項目を一括して確認している
- 挙動確認・課題管理表へ判定と証拠を反映している

重大な回帰が見つかった場合は、新規実装を止め、原因を含む最も近い制作フローから修正する。
